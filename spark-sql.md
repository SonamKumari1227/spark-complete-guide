# Spark SQL

## Introduction

Imagine you're working with a dataset containing hundreds of millions of records stored in Parquet files inside a Data Lake.

You need to answer questions such as:

- How many customers placed orders last month?
- What is the total revenue generated by each region?
- Which products contributed the most sales?

One approach is to write low-level Spark transformations:

```python
orders_df.groupBy("region") \
         .agg(sum("revenue"))
```

While this works, many Data Engineers, Data Analysts, and BI teams are already familiar with SQL.

To bridge the gap between distributed computing and traditional SQL-based analytics, Spark introduced **Spark SQL**.

Spark SQL allows developers to process massive datasets using familiar SQL syntax while still leveraging Spark's distributed execution engine.

---

# What is Spark SQL?

Spark SQL is a Spark module used for processing structured and semi-structured data using:

- SQL Queries
- DataFrame APIs
- Dataset APIs (Scala/Java)

It provides a unified interface for working with:

- Parquet Files
- Delta Tables
- JSON Files
- CSV Files
- Hive Tables
- Iceberg Tables
- Relational Databases

Instead of writing complex distributed processing logic, users can express business logic using SQL.

Example:

```sql
SELECT department,
       AVG(salary)
FROM employees
GROUP BY department;
```

Spark automatically converts this query into distributed tasks executed across the cluster.

---

# Why Spark SQL Was Introduced

Before Spark SQL, most Spark workloads relied on RDDs.

Example:

```python
rdd.filter(lambda x: x.salary > 50000)
```

The problem was that Spark had no understanding of:

- Column Names
- Data Types
- Relationships
- Query Intent

As a result:

- No Query Optimization
- Poor Performance for Analytics
- Complex Development

Spark SQL introduced structured data processing.

This enabled:

- Schema Awareness
- Query Optimization
- SQL Support
- Better Performance

# Core Components of Spark SQL

Spark SQL consists of several important components.

## 1. DataFrame API

The DataFrame API provides a programmatic way to interact with structured data.

Example:

```python
employees_df.filter(
    col("salary") > 50000
)
```

Internally, Spark converts DataFrame operations into SQL-like execution plans.

---

## 2. SQL Interface

Spark allows users to execute standard SQL queries.

Example:

```sql
SELECT *
FROM employees
WHERE salary > 50000;
```

This makes Spark accessible to SQL developers and analysts.

---

## 3. Catalyst Optimizer

One of Spark SQL's most powerful features.

Catalyst analyzes queries and automatically optimizes them before execution.

Example:

```sql
SELECT *
FROM employees
WHERE department = 'IT'
```

Catalyst may perform:

- Predicate Pushdown
- Constant Folding
- Projection Pruning
- Join Reordering

without requiring code changes.

---

## 4. Tungsten Execution Engine

Tungsten is responsible for efficient execution.

It improves:

- CPU Utilization
- Memory Management
- Binary Processing
- Code Generation

This significantly improves query performance.

---

# How Spark SQL Works

Consider the following query:

```sql
SELECT department,
       COUNT(*)
FROM employees
GROUP BY department;
```

Spark processes the query as follows:

```text
SQL Query
      ↓
Parse Query
      ↓
Logical Plan
      ↓
Optimized Logical Plan
      ↓
Physical Plan
      ↓
Task Generation
      ↓
Execution on Executors
```

---

# Query Lifecycle

## Step 1: Parsing

Spark converts SQL text into an abstract syntax tree.

```sql
SELECT * FROM employees
```

becomes:

```text
Parsed Query Tree
```

---

## Step 2: Logical Plan

Spark creates a logical representation of the query.

Example:

```text
Project
  ↓
Filter
  ↓
Table Scan
```

No execution happens yet.

---

## Step 3: Optimization

Catalyst applies optimization rules.

Examples:

- Push filters closer to source
- Remove unnecessary columns
- Reorder joins

Result:

```text
Optimized Logical Plan
```

---

## Step 4: Physical Planning

Spark generates multiple physical execution strategies.

Examples:

- Broadcast Hash Join
- Sort Merge Join
- Shuffle Hash Join

Catalyst selects the most efficient plan.

---

## Step 5: Execution

Tasks are generated and distributed across executors.

```text
Driver
   │
   ▼
Tasks
   │
   ▼
Executors
```

# Creating DataFrames

Spark SQL primarily works with DataFrames.

Example:

```python
employees_df = spark.read.csv(
    "employees.csv",
    header=True,
    inferSchema=True
)
```

---

# Creating Temporary Views

To execute SQL queries, DataFrames can be registered as temporary views.

Example:

```python
employees_df.createOrReplaceTempView(
    "employees"
)
```

Now Spark can query it using SQL.

---

# Running SQL Queries

Example:

```python
result = spark.sql("""
SELECT department,
       AVG(salary)
FROM employees
GROUP BY department
""")
```

---

# Temporary Views vs Global Temporary Views

## Temporary View

Visible only within the current SparkSession.

```python
df.createOrReplaceTempView(
    "employees"
)
```

---

## Global Temporary View

Accessible across Spark sessions.

```python
df.createGlobalTempView(
    "employees"
)
```

Query:

```sql
SELECT *
FROM global_temp.employees
```

# Spark SQL and DataFrames

A common interview question is:

> Are DataFrames and Spark SQL different?

The answer is:

> No. They are two different ways of expressing the same logic.

Example:

### DataFrame API

```python
employees_df.filter(
    col("salary") > 50000
)
```

### Spark SQL

```sql
SELECT *
FROM employees
WHERE salary > 50000
```

Both produce similar execution plans.

# Spark SQL and Joins

Spark supports all major SQL joins.

---

## Inner Join

```sql
SELECT *
FROM customers c
INNER JOIN orders o
ON c.customer_id = o.customer_id
```

---

## Left Join

```sql
SELECT *
FROM customers c
LEFT JOIN orders o
ON c.customer_id = o.customer_id
```

---

## Right Join

```sql
SELECT *
FROM customers c
RIGHT JOIN orders o
ON c.customer_id = o.customer_id
```

---

## Full Outer Join

```sql
SELECT *
FROM customers c
FULL OUTER JOIN orders o
ON c.customer_id = o.customer_id
```

# Adaptive Query Execution (AQE) and Spark SQL

Modern Spark SQL workloads benefit heavily from AQE.

AQE can:

- Dynamically switch join strategies
- Handle data skew
- Merge small shuffle partitions

Example:

```text
Sort Merge Join
       ↓
AQE
       ↓
Broadcast Hash Join
```

This happens automatically during runtime.

|----------|----------|----------|

| Big Data Support | Limited | Massive Scale |
| Parallel Execution | Limited | Yes |
| Fault Tolerance | Depends | Yes |
| Query Optimization | Yes | Yes (Catalyst) |
| Storage Formats | Database Tables | Multiple Formats |
